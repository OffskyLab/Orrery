orrery enter / exit 動詞設計

2026-05-22 · feat/v3-command-restructure · brainstorm 完成設計
用兩個 top-level 動詞 orrery enter <sandbox> 和 orrery exit 取代 orrery sandbox use <name> — 把「進入 sandbox」(opt-in) 和「回到 origin」(opt-out) 當成兩個對等但相反的動作,而不是塞進同一個 verb,讓 origin 永遠是「沒進 sandbox」的狀態而不是某種叫 origin 的 sandbox。

1. 為什麼要動

v3 草案

一個動詞兼差兩件事

orrery sandbox use S-A — 進入 sandbox

orrery sandbox use origin — 回到 origin

↳ origin 被當成一種「叫 origin 的 sandbox」, 跟使用者心智模型「origin = 我真正的 config」不合。

↳ Hint string 要嘛印 orrery sandbox use origin 這種彆扭句, 要嘛假裝沒這個 case。

→
新設計

對稱的兩個動詞

orrery enter S-A — 進入 sandbox

orrery exit — 離開回 origin

↳ 「進入」跟「離開」對稱,使用者直覺。

↳ Origin 變回「狀態」, 不再是某個 sandbox 的名字。

↳ v3 還沒 ship, 砍 sandbox use 零相容性成本。

2. 狀態機

進入 退出 透明切換 no-op 拒絕

3. 行為矩陣

輸入目前狀態結果動作
orrery enter X origin OK export X · ORRERY_ACTIVE_ENV=X · 印 switched 訊息
orrery enter X sandbox Y 透明切換 unexport Y · export X · ORRERY_ACTIVE_ENV=X · 印 switched 訊息
orrery enter origin 任何 拒絕 stderr: 'origin' is not a sandbox — use 'orrery exit' to leave the current sandbox. exit 1
orrery enter <not-exist> 任何 錯誤 stderr: sandbox '<name>' not found exit 1（同現行 sandbox use 錯誤）
orrery exit sandbox X OK unexport X · unset Claude/Codex/Gemini env vars · ORRERY_ACTIVE_ENV=origin · 寫 current=origin · 印 switched-to-origin 訊息
orrery exit origin no-op stderr: Already at origin (no sandbox active). exit 0

4. CLI surface 變化

Top-level（OrreryCommand subcommands）

enter exit add list show use remove sandbox tools which run resume delegate sessions setup init update uninstall install

加上 enter、exit 兩個新 verb；其他不動。

orrery sandbox subcommands

use create list delete info rename current set-env unset-env memory sync export unexport

砍掉 sandbox use；其他 12 個 subcommand 留著。

5. Shell function dispatch（top-level case）

v3 草案（已 commit）

orrery() {
case "$cmd" in
sandbox)
case "${2:-}" in
use) [44 行 env-var dance + origin branch]
create) ...
*) command orrery-bin "$@"
esac ;;
run) ...
add) ...
*) command orrery-bin "$@"
esac
}

新設計

orrery() {
case "$cmd" in
enter) [原 sandbox/use 內容 + 拒絕 enter origin]
exit) [原 sandbox/use 的 origin 分支 + no-op 提醒]
sandbox)
case "${2:-}" in
create) ...
*) command orrery-bin "$@"
esac ;;
run) ...
add) ...
*) command orrery-bin "$@"
esac
}

需要連帶更新的兩個 call site

位置舊新
_orrery_init 自動啟用 orrery sandbox use "$env_name" orrery enter "$env_name"
run) case 內 phantom-claude target 預設 orrery sandbox use "$_run_target" orrery enter "$_run_target"

6. Phantom 流程

Sentinel 不變、slash command 不變 — 只在 supervisor 端做一次 origin 翻譯。

關鍵： PhantomSandboxTriggerCommand 本身不動 — 它寫 sentinel 的方式照舊。 唯一的翻譯邏輯放在 shell function 的 phantom-loop 裡，集中、好驗證、也不會洩漏到使用者面前。

9. 範圍

範圍內

  • 新增 orrery enter / orrery exit top-level commands
  • 刪除 orrery sandbox use（CLI + shell + L10n）
  • L10n key 重命名 use.* → enter.* / exit.*
  • 所有指向 orrery sandbox use 的 hint 字串全換
  • Phantom supervisor 加 TARGET_SANDBOX=origin 翻譯
  • _orrery_init + run) 兩處 call site 改用 enter
  • 對應 tests 更新
  • keys.md doc 對齊

不在範圍

  • 不重命名 env var ORRERY_ACTIVE_ENV（牽連 statusline / MCP / 外部 consumer）
  • 不重命名 Swift 內部型別 EnvironmentStore / Environment / ReservedEnvironment
  • 不動 SandboxCommand 其他 12 個 subcommand
  • 沒有 sandbox use compat alias / hidden command（v3 未 ship、乾淨切）
  • 不改 PhantomSandboxTriggerCommand 自身 sentinel 寫法
  • 不改 /orrery:phantom slash command 介面

10. 風險與回滾

Pre-ship、無外部依賴： v3 還沒釋出，沒人在用 sandbox use。若實作後發現方向不對, 回 revert 即可、spec 保留作為紀錄。 Shell function 在每次 eval "$(orrery init)" 重新生成 — 同一台機器不會卡在「半套用」狀態。
