orrery rc.2 設計：Workspace Layout 統一

把「sandbox/env」與 v3.1「workspace」兩個撞名的概念合併成單一 workspace，並讓 origin 成為零特例的成員——使用者原本的 ~/.claude 就是 origin workspace。

日期：2026-05-29　·　目標版本：v3.1.0-rc.2（RC，僅供自我驗證，不給真實使用者）　·　狀態：設計待 review

origin（takeover） workspace（共享內容） account（身份） symlink 要消滅的東西

1 · 問題：一個名字，三個身體

v3.1 引入了兩個都叫「workspace」的東西，加上 origin 的特例處理，結果 origin 同時存在於三個地方，彼此看不到對方的 session。

sandbox / env

隔離單位：一組環境變數 + 各工具 config dir。rc.1 把使用者面詞彙 rename 成 “workspace”。
PATH
~/.orrery/envs/<UUID>/{claude,codex,gemini}/

v3.1 “workspace”

claude 的共享內容命名空間（projects/memory/…），pin 的 account symlink 指進來。
PATH
~/.orrery/envs/<UUID>/claude-workspace/

origin（takeover）

使用者真實的 ~/.claude，被接管進來。但 rc.1 又另外建了一個 envs/origin/claude-workspace/。
PATH
~/.orrery/origin/claude/
＋ envs/origin/claude-workspace/ ⟵ 多餘
後果 用 plain claude 寫的 session（在 takeover root）跟 orrery use <origin-pinned-account> 寫的 session（在 envs/origin/claude-workspace/）互相看不到——明明都叫 “origin”，卻是兩個物理位置。程式碼裡到處是 if workspace == "origin" 的特例分支。

根本洞察

沒有兩個概念。workspace 就是那個隔離單位——它同時包含「給 codex/gemini 的環境變數/config」和「給 claude 的共享內容」。origin 只是那個物理根目錄等於 takeover root 的預設 workspace：使用者既有的 ~/.claude 變成 origin workspace。從來就不該為它另外開一個 v3.1 workspace 目錄。

2 · 目標：一種 layout，origin 零特例

# 統一後：workspace = 一個 root，底下放 claude/codex/gemini
~/.orrery/
  workspaces/                       （從 envs/ 改名）
    origin/                         保留名目錄 = takeover workspace
      claude/                       ← ~/.claude symlink 指這裡（takeover root）
      codex/                        ← CODEX_HOME / CODEX_CONFIG_DIR
      gemini/                       ← GEMINI_CONFIG_DIR
      workspace.json                metadata（取代 origin/config.json）
    <UUID>/                         named workspace；目錄名「一律」UUID
      claude/                       projects/ memory/ agents/ commands/ todos/
      codex/  gemini/
      workspace.json                { "name": "work", … } 顯示名，可重複
  accounts/
    claude/<id>/                    per-account dir = CLAUDE_CONFIG_DIR
      projects/ → ../../../workspaces/<pinned>/claude/projects
      memory/   → …                 （agents commands todos 同理）
      metadata.json                 Account model（含 workspace 欄位）
  shared/                           不動 — claude session jsonl 實體存這，
                                    workspaces/<ws>/claude/projects symlink 指進來

關鍵不變式

  • workspace 目錄名永遠是 UUID，只有保留字 origin 例外。顯示名放在 workspace.json.name，可自由重複——UUID 負責區分。（開發機看到的 personal/、work/ 等非 UUID 目錄是上線前的 0.x 遺留，真實使用者沒有。）
  • origin 是一般 workspace，結構與 named 完全相同（<root>/{claude,codex,gemini}/ + workspace.json）。唯一差別：保留名 origin + 它的 claude/ 是 takeover root。不再有 if workspace == "origin" 路徑分支，只剩解析 workspace 時的保留名檢查。
  • account.workspace 存 UUID 或字面 "origin"。account dir symlink 解析到 workspaces/<account.workspace>/claude/<subdir>。
  • codex/gemini 沒有 per-account dir，用整包 config dir，由 workspace enter 時的環境變數指向。

account dir（per-account）

accounts/claude/<id>/projects
accounts/claude/<id>/memory
accounts/claude/<id>/agents
…/commands　…/todos
→

pinned workspace 的 claude/

workspaces/<pinned>/claude/projects
workspaces/<pinned>/claude/memory
workspaces/<pinned>/claude/agents
…/commands　…/todos

3 · 路徑常數 before / after

EnvironmentStore.swift 的路徑常數變更。注意 origin 改到 workspaces/ 底下後，originDir 可用與 named workspace 相同的 base 表達，正是零特例的底層體現。

常數BeforeAfter
envsURL~/.orrery/envs/~/.orrery/workspaces/
originDir~/.orrery/origin/~/.orrery/workspaces/origin/
originConfigURLorigin/config.jsonworkspaces/origin/workspace.json
claudeWorkspaceDir(ws)envs/<ws>/claude-workspace/workspaces/<ws>/claude/
originConfigDir(tool)origin/<tool>/workspaces/origin/<tool>/
toolConfigDir(tool, env)envs/<id>/<tool>/workspaces/<id>/<tool>/
sharedURL / shared* / accounts*不變
claudeWorkspaceDir(ws) 變更後對 named 與 origin 都回傳 workspaces/<ws>/claude/——不再有獨立的 “claude-workspace” 子目錄。takeover root 的 claude/ 就是 origin workspace 的 claude 內容目錄。

4 · Metadata 統一：兩個 model 合一

OriginConfig（origin/config.json）與 OrreryEnvironment（env.json）欄位幾乎相同。合併成單一 Workspace model，序列化為 workspace.json。

Before：兩個 struct

OriginConfig → origin/config.json
　isolateMemory, memoryStoragePath,
　isolatedSessionTools, accounts

OrreryEnvironment → env.json
　＋ id, name, description,
　createdAt, lastUsed, tools, env

After：單一 Workspace

Workspace → workspace.json
　id（UUID 或 "origin"）, name,
　description, createdAt, lastUsed,
　tools, env, isolatedSessionTools,
　isolateMemory, memoryStoragePath,
　accounts[Tool.rawValue → AccountID]
  • 移除 OriginConfig；origin 讀寫同一個 Workspace model。
  • 解碼寬容：缺新欄位的 workspace.json（即遷移自舊 config.json）把 id/name 預設為 "origin"。
  • 型別 OrreryEnvironment → Workspace、檔名 env.json → workspace.json 全碼改名。OrreryEnvironment.defaultName（"origin"）→ Workspace.reservedOriginName。

5 · Migration：兩階段，順序是關鍵

把任何真實使用者的狀態（v3.0.x 的 envs/<UUID>/ + accounts/ pool + origin/）冪等收斂到 workspaces/ layout。RC 從未給真實使用者，所以 rc.1 的 claude-workspace/（與誤建的 envs/origin/）中間態直接忽略——不為不存在的狀態加補償層。rc.1 那支 runV31AccountLayoutIfNeeded 在源頭被正確版本取代，而非疊層修補。

為什麼必須拆兩階段 單一 migration 放在 main.swift chain 的任一位置都會錯，因為兩個依賴互相矛盾：
① 結構搬遷（改名、搬 takeover root、重指 ~/.claude）必須在 OriginTakeoverBootstrap 之前——否則 takeover 用新的 originDir 常數（workspaces/origin/）動作，但實體還在 origin/、~/.claude 還指舊位置，takeover 誤判並跟還沒跑的遷移打架。
② account symlink 重指依賴 account pool 已存在，而那是 AccountMigration.runIfNeeded（v2→v3）在 chain 中較晚才建的。

於是 main.swift 的遷移鏈變成（兩階段皆 flag-guarded、best-effort、永不 throw）：

1
LegacyOrbitalMigration.runIfNeeded()
orbital → orrery 改名遷移（既有）
2
runWorkspaceStructureRelocationIfNeeded()Phase A · 新
envs/→workspaces/；origin/→workspaces/origin/（若目標已存在則不覆蓋 takeover root，log 衝突）；重指 ~/.claude；config.json/env.json→workspace.json；claude-workspace/ 併入 claude/。flag：.workspace-structure-relocated
必須在 takeover 之前：先把實體路徑就位，後續用新常數的 code 才正確
3
OriginTakeoverBootstrap.runIfNeeded()
此時 isOriginManaged==true（或對全新使用者乾淨 takeover 進新位置）→ no-op-或-正確，不打架
4
AccountMigration.runIfNeeded()
v2→v3 account-pool 遷移（建立 pool）
5
runInfoBackfillIfNeeded()
email/plan 事後補齊（既有）
6
runWorkspaceAccountSymlinksIfNeeded()Phase B · 取代 rc.1
每個 claude account：重建 5 條 symlink 指向 workspaces/<account.workspace>/claude/<subdir>。flag：.workspace-account-symlinks
必須在 account pool 存在之後
不認得的目錄 → 留著不動 + log 非 UUID 名目錄、或缺 env.json/workspace.json 的 UUID 目錄：兩階段都原樣保留並記錄。Migration 絕不刪除使用者目錄——為真實使用者不可能有的狀態承擔不可逆刪除風險不值得。開發機殘骸由獨立手動腳本清理。

6 · Uninstall round-trip：免費的好處

originRelease / originTakeover / isOriginManaged 自動跟隨變更後的 originConfigDir 常數 → 從 workspaces/origin/<tool>/ release 回 ~/.claude。

結構性消除孤兒 session 因為 origin 的 claude 內容（含 v3.1 wrapper 寫入的 session）現在一直都在 takeover root 裡，uninstall 既有的 release 邏輯把所有東西折回 ~/.claude/projects/，不需要任何額外的 merge-back 程式碼。rc.1 的「孤兒 v3.1 session」問題從結構上消失。

8 · 範圍：明確的進與不進

✗ 明確延後

  • 使用者面指令詞彙收斂（enter/exit/use/pin/workspace）— 獨立 UX 設計
  • 「sandbox = (workspace, account) 高級封裝」— 自己的 brainstorm

  • 反向遷移（workspaces→envs）— 維持單向

9 · 測試策略

  • Unit：Workspace model encode/decode，含寬容解碼舊 config.json（缺 id/name）與舊 env.json。
  • Migration（temp ORRERY_HOME）：給定合成的 v3.0.x 樹，斷言 — workspaces/ 在、envs/ 消失；workspaces/origin/ 在、origin/ 消失；每個 workspace.json 可解碼、claude-workspace/ 不存在；account symlink 解析到 workspaces/<ws>/claude/<subdir>；兩個 flag 都寫、各 phase 二次執行為 no-op；Phase A 在 takeover 前不打架（chain 後 isOriginManaged）、Phase B 在 pool 存在後；不認得的目錄留著 + log（不刪）。
  • 冪等：跑兩次 migration 得到同一棵樹。
  • Uninstall round-trip：takeover → 經 wrapper 寫一個 session → uninstall → 斷言 session 內容回到 ~/.claude/projects/。
  • L10n build gate 維持綠（all-locales-have-all-keys）；完整 swift test（315 baseline）通過。
