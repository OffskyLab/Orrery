# Magi 角色分化與決策壓力機制

## 議題定義

### 背景

目前 Magi 的三個模型（Claude、Codex、Gemini）在討論中各自提供分析，但觀察角度未被明確區分。如果三個模型都從相似的角度思考，討論容易趨於同質化，降低了多模型協作的價值。

核心假設：**若每個模型被賦予不同的觀察角度與決策壓力，討論的完整度會顯著提高，進而產出更完善的 spec 和更高成功率的實作。**

### 目標

1. **提升討論完整度** — 透過角色分化，確保每次討論涵蓋更多面向（技術可行性、使用者體驗、架構影響等）
2. **產出更完善的 spec** — 討論結論能直接轉化為高品質的規格書，減少 spec 階段的漏洞
3. **提高實作驗證成功率** — 因為討論已從多角度檢驗，實作時遇到未預期問題的機率降低

### 範圍

**討論內：**
- 三個模型各自應扮演什麼角色 / 觀察角度
- 如何設計「決策壓力」機制（讓每輪討論更有效率地推向結論）
- 角色分化如何反映在 prompt 設計和討論流程中
- 討論產出如何更好地銜接 spec 撰寫

**討論外：**
- Magi 的基礎架構變更（MCP tool、並行化等，已有獨立討論）
- 模型選擇或替換

### 約束

- 角色設計需兼顧靈活性 — 不同議題可能需要不同的角度組合
- 決策壓力不應犧牲討論深度 — 壓力是為了聚焦，不是為了草率結論
- 需與現有 `/discuss` 和 `/write-spec` 流程相容

### 子議題

- [ ] 子議題 1：角色定義 — 三個模型各自的觀察角度應該是什麼？（固定角色 vs. 動態分配）
- [ ] 子議題 2：決策壓力機制 — 如何在每輪討論中施加壓力推動收斂？（投票、deadline、devil's advocate 等）
- [ ] 子議題 3：Prompt 設計 — 如何在 system prompt 中體現角色分化和決策壓力？
- [ ] 子議題 4：討論→Spec 銜接 — 如何讓分化後的討論產出更直接地轉化為 spec 內容？
